# Data Sources and Generated Outputs

This document defines which project files are authoritative inputs, which files
contain project-owned rules, and which files are generated. Generated files
must be reproducible from the inputs and should not become independent sources
of truth.

## Authority order

When two sources describe the same value, use this order:

1. The active ruleset source selected by `src/rulesets/sources.js`.
2. Project-owned auxiliary rule sources for gaps the ruleset does not carry.
3. Generated normalized data and browser bundles.
4. UI state, saved rosters, and exported rosters.

Project-owned auxiliary rules should replace or extend a specific imported
rule. They must retain source/provenance and should not erase the imported
value they are supplementing.

## Authoritative inputs

### Ruleset registry

Primary editable source:

```text
src/rulesets/sources.js
```

The registry defines available rulesets, their source paths, parser format,
primary/default status, and auxiliary rule inputs. The current default ruleset
is:

```text
wh40k-11e-vflam
```

Its imported source lives under:

```text
data/rulesets/wh40k-11e-vflam/
```

That directory is a local imported copy of BSData-style 11th-edition JSON
catalogues. Treat it as replaceable imported data: update or replace the
source, then regenerate the browser bundle.

### Imported BSData

Location:

```text
data/wh40K/wh40k-10e-main/wh40k-10e-main/
```

The `.cat` catalogue files and `Warhammer 40,000.gst` are the original
10th-edition BSData scaffold. They remain useful for historical parser tests
and comparison, but they are not the current default builder source.

Both the 10e XML scaffold and the active 11e JSON source contain more than flat
datasheet values:

- selectable units and shared definitions
- stable IDs and cross-catalogue links
- base point costs
- conditional point modifiers
- unit/model minimum and maximum selections
- model and wargear choices
- selection constraints and conditions
- profiles, characteristics, weapons, abilities, and rule links
- categories, keywords, and faction relationships

Imported ruleset files should be treated as read-only inputs. Updating a ruleset
means replacing/updating its source snapshot, then regenerating downstream
files.

### Auxiliary 11e rule sources

Primary editable sources:

```text
data/manual-rules/wh40k-11e-core-stratagems.json
data/manual-rules/wh40k-11e-tyranids-detachment-stratagems.json
data/manual-rules/wh40k-11e-wahapedia-detachment-stratagems.json
data/rulesets/wh40k-11e-newrecruit/stratagems.json
```

The active 11e catalogue source has units, points, loadouts, detachments,
enhancements, leader relationships, and constraints, but does not currently
carry complete structured detachment stratagem records. The registry layers
these auxiliary sources in after army extraction.

Manual files should be small, source-linked, and additive. Prefer filling gaps
without replacing imported or New Recruit records that already exist.

### Legacy 11th points workbook

Historical editable source:

```text
data/overrides/11th/warhammer_points_changes_by_faction.xlsx
```

This workbook was used while the project still treated 10e data as scaffolding
with an 11e override layer. It is no longer the preferred path for active 11e
ruleset work, but it remains in the repo for reference and migration checks.

Generated parsed override:

```text
data/overrides/11th/points-overrides.json
```

This JSON file is generated by:

```text
scripts/build-points-overrides-from-xlsx.js
```

Do not manually maintain the JSON when the workbook is the upstream source.

## Transformation pipeline

### Active ruleset normalization

Generator:

```text
src/rulesets/sources.js
src/bsdata/unit-definitions.js
src/bsdata/army-definitions.js
src/bsdata/ally-definitions.js
```

The default pipeline extracts the active registry source into normalized unit,
army, and ally definitions. It supports both XML `.cat/.gst` and JSON catalogue
shapes through the shared BSData normalization layer.

### Browser engine bundle

Generator:

```text
scripts/build-engine-ui-data.js
```

Generated output:

```text
ui/engine-data-milestone15.js
```

This is the browser payload used by the current UI. It is generated data and
must not become an independent rules source.

### Legacy generated reports

```text
scripts/export-faction-roster-units.js
scripts/export-all-roster-units.js
scripts/normalize-roster-units.js
scripts/export-faction-datasheets.js
scripts/export-all-factions.js
```

Generated outputs include:

```text
data/roster-rules/*-roster-units.json
data/builder-units/*-builder-units.json
data/json/*-datasheets.json
```

These files are useful diagnostics and historical extraction artifacts. They
are not the current browser source of truth.

### Builder rule classification

Generators include:

```text
scripts/extract-builder-rules.js
scripts/analyze-builder-rules.js
scripts/apply-leader-attachments.js
scripts/resolve-leader-targets.js
```

Generated output:

```text
data/builder-rules/*-builder-rules.json
data/builder-rules/leader-attachments-raw.json
```

These files are derived interpretations of BSData rules and leader links. They
should remain reproducible reports/indexes rather than editable rule sources.

## Hand-written application and tooling

These are project source code and should be edited directly:

```text
src/**/*.js
scripts/*.js
scripts/leader/*.js
ui/index.html
ui/engine-app.js
ui/engine-runtime.js
ui/styles.css
package.json
```

Files in the project root such as `*-debug.*`, `*-dump.txt`, `cat-files.txt`,
and `json-files.txt` are diagnostic artifacts. They are neither rules sources
nor required runtime inputs.

`node_modules/` is installed dependency output and is not project source.

## Worked example: Arco-Flagellants

The 10e Adepta Sororitas BSData catalogue contains all of the following:

- unit ID `ca59-3760-5efc-9846`
- base cost `45`
- one model selection with minimum `3` and maximum `10`
- a conditional modifier setting the unit cost to `140` when at least four
  Arco-Flagellant models are selected

The current normalized engine preserves conditional modifiers structurally. The
desired flow remains:

```text
Ruleset definition (sizes + base/conditional prices)
  -> normalized unit definition
  -> auxiliary rules or overrides when explicitly configured
  -> configured roster entry price
```

It should not be:

```text
one flattened BSData price
  -> spreadsheet text heuristics
  -> UI-created size options
```

## Rules for future changes

- Preserve BSData IDs through every generated stage.
- Preserve conditions and modifiers structurally; do not reduce them to one
  point value during extraction.
- Keep unit definitions separate from configured roster entries.
- Attach auxiliary rules to stable IDs when possible, with normalized-name
  matching used only as a migration fallback.
- Record the origin of every effective price and rule: imported base,
  conditional modifier, auxiliary source, or named override.
- Never manually repair a generated faction JSON file. Fix its generator or
  upstream source and regenerate it.
